# VIP-0002: Message-Size Profile Plumbing

**Status:** Draft — data model and read paths exist; observation, persistence, defaults, and
update APIs are **not implemented**. Tracked work items in
[Implementation Work Items](#6-implementation-work-items).

**Related:** [VIP-0001](VIP-0001.md) §8 (dimension coupling), §11.1 (topic entities), §13
(failure handling), §19 (open questions).

---

## 1. Summary

Each topic may carry a `MessageSizeProfile` — observed `avgMsgSizeBytes` and `maxMsgSizeBytes`
— to couple the qps and bytes rate-limit dimensions, validate capacity at creation, and support
capacity planning.

Today the field is **optional plumbing only**: it can be supplied at topic create and is
persisted on `VaradhiTopic`, but nothing observes produce traffic, nothing updates it when
message patterns change, and no deployment default applies when it is absent.

---

## 2. Goals & Non-Goals

### Goals
- Persist per-topic `avg` / `max` message size (payload + headers, `Message.getTotalSizeBytes()`).
- Feed the rate limiter's bytes-budget floor (`bytesQuota ≥ qpsQuota × maxMsgSize`) per VIP-0001 §8.
- Validate at topic write that `throughputKBps` is consistent with `qps × maxMsgSize`.
- Apply a deployment-default size assumption when a topic has no profile yet.
- Refresh per-pod quotas when the profile changes (without pod restart).

### Non-Goals (v1 of this VIP)
- **Billing or metering** from the profile — guard-rail input only.
- **Coordinator transport of live avg/max** — deferred with VIP-0001 §18 adaptive phase.
- **Automatic topic capacity resizing** from observed sizes — operators may use the data for
  planning; auto-mutation of `TopicCapacityPolicy` is out of scope.

## 3. Background & Current State

### 3.1 Entity model

| Type | Role |
|---|---|
| `MessageSizeProfile` | Immutable value object: `avgMsgSizeBytes`, `maxMsgSizeBytes` (avg clamped to max). |
| `VaradhiTopic.messageSizeProfile` | Persisted field; `null` when unset. |
| `TopicResource.messageSizeProfile` | Optional API/factory input on create; round-tripped on GET. |

Javadoc describes the profile as *observed* sizes, but no observation loop exists yet.

### 3.2 How it is populated today

1. **Topic create** — `POST /v1/projects/:project/topics` deserializes `TopicResource`; if
   `messageSizeProfile` is present, `VaradhiTopicFactory` validates capacity consistency and
   copies it onto `VaradhiTopic`.
2. **Factory defaults** — unlike `capacity` and `rateLimiterMode`, the factory does **not**
   default `messageSizeProfile`; omitted → `null`.
3. **Metastore** — serialized on `VaradhiTopic` in ZK; legacy JSON without the field
   deserializes with `null` (`VaradhiTopicSerializationTest`).
4. **No update path** — `TopicHandlers` exposes create, get, delete, restore only; no PATCH/PUT
   for profile or other auxiliary fields.

### 3.3 Consumers (read-only)

| Consumer | Use |
|---|---|
| `VaradhiTopicFactory` (create) | Reject if `throughputKBps < qps × maxMsgSize`; warn if tight for avg. |
| `EvenSplitPerPodTopicQuotaProvider` | Dimension coupling: `bytesQuota = max(bytesQuota, qpsQuota × maxMsgSize)`. |
| `TopicResource.from()` | GET response round-trip. |

The produce hot path does **not** read or write the profile.

### 3.4 Open code TODO

`VaradhiTopic` carries a TODO on where topic auxiliary data (including `messageSizeProfile`)
should live long-term — secondary to CRUD/produce/consume but used by rate limiting and
capacity planning.

## 4. Behaviour When Profile Is Not Initialized

Rate limiting **still runs** when `messageSizeProfile` is `null`. A missing profile does **not**
disable the limiter.

What changes:

| Aspect | With profile | Without profile (`null`) |
|---|---|---|
| QPS bucket | Normal (`qps` split across pods) | Same |
| Bytes bucket | Normal (`throughputKBps` split; debit actual message size) | Same |
| Dimension coupling | `bytesQuota` floored at `qpsQuota × maxMsgSize` | **Skipped** (`maxMsgSize = 0`) |
| Create-time validation | `throughputKBps ≥ qps × maxMsgSize` enforced | **Skipped** |
| Deployment default | N/A | **Not implemented** (VIP-0001 §11.1 intent only) |

**Practical effect:** for **low QPS + large messages**, without the floor the bytes bucket
capacity may be smaller than one max-sized message, so large messages are more likely to be
throttled than VIP-0001 §8 intends. For **high QPS + small messages**, qps is usually binding
and impact is small.

All other rate-limit behaviour is unchanged: deployment `enabled` kill switch, per-topic
`rateLimiterMode` (`disabled` / `shadow` / `enforced`), pod even-split, `minPodQps` floor,
fail-open on check errors.

## 5. Design Intent (VIP-0001) vs Actual

**Intended (VIP-0001 §8, §289–294, §11.1):**

- Producers observe message size; the system persists `avg` / `max` on the topic.
- Each pod caches the profile with topic config and applies the bytes-capacity floor locally.
- Topic creation uses the profile to validate `qps ↔ throughputKBps` consistency.
- Until a topic has a profile, fall back to a **deployment-default size assumption**.

**Actual today:**

- Profile is operator-supplied at create only (optional JSON field).
- If omitted → `null` forever (no auto-default, no auto-observation).
- Rate limiter treats `null` as no max-size floor.
- Produce pattern changes have no effect on the stored profile.

## 6. Implementation Work Items

Use these as GitHub issues (epic + sub-issues or standalone).

### 6.1 Observe per-topic message sizes on produce

**Title:** Observe per-topic message sizes on produce and maintain avg/max profile

Sample `Message.getTotalSizeBytes()` on the produce hot path (per topic). Maintain rolling or
windowed avg and max (algorithm TBD). Thread-safe, low-allocation; no blocking I/O on hot path.

**Acceptance criteria:**
- Per-topic avg/max available in-memory on each producer pod.
- Observable via metrics/logging for validation.
- No produce-path regression.

**Refs:** `MessageSizeProfile`, `ProduceHandlers`, `ProducerService`, VIP-0001 §8, §289–294.

---

### 6.2 Persist observed profile to metastore

**Title:** Persist observed message-size profile on `VaradhiTopic` in metastore

Async persistence of updated `avgMsgSizeBytes` / `maxMsgSizeBytes`. Debounce/coalesce writes.
Emit resource change event for `ResourceReadCache` invalidation. Concurrent multi-pod update
policy TBD (last-writer or merge).

**Acceptance criteria:**
- Updated profile visible via `GET /v1/projects/:project/topics/:topic`.
- Other pods pick up changes via cache invalidation.
- Legacy topics without profile continue to work.

**Refs:** `VaradhiTopic`, `TopicStore`, `ResourceReadCache`, `VaradhiTopicSerializationTest`.

---

### 6.3 Deployment default when profile is absent

**Title:** Define and implement deployment-default message size when topic has no profile

Add config (e.g. in `RateLimiterOptions`): default `avg` and/or `max` msg size bytes. Apply at
quota computation when `topic.getMessageSizeProfile() == null`. Optionally apply at topic
creation validation. Document conservative vs permissive semantics.

**Acceptance criteria:**
- Topics without profile get dimension coupling using deployment default.
- Config validated at startup.
- Tests for null-profile topics with default enabled.

**Refs:** `EvenSplitPerPodTopicQuotaProvider`, VIP-0001 §11.1, §435, §640–641.

---

### 6.4 Topic update API for profile and RL metadata

**Title:** Add topic update API for message-size profile and rate-limit metadata

`PATCH` or `PUT` on `/v1/projects/:project/topics/:topic` for auxiliary fields:
`messageSizeProfile`, optionally `perRegionQuotaWeights`, `rateLimiterMode`. Reuse factory
validation. Immutable entity rebuild + `topicStore.update` + resource event. Authz:
`TOPIC_UPDATE`.

**Acceptance criteria:**
- Operator can set/update profile without recreating topic.
- Validation rejects inconsistent capacity vs profile.

**Refs:** `TopicHandlers`, `TopicResource`, `VaradhiTopicService`.

---

### 6.5 Recompute rate-limit quotas when profile changes

**Title:** Recompute per-pod quotas when topic message-size profile changes

`ProduceRateLimiter` refreshes quotas on pod-count change but not on topic config updates. On
topic resource update event, mark quota stale for that topic; `refreshQuotaIfStale` recomputes
via `quotaProvider.quotaFor(topic)`.

**Acceptance criteria:**
- Profile update changes bytes-budget floor without pod restart.
- Test: update profile → bytes quota reflects new `maxMsgSize`.

**Refs:** `ProduceRateLimiter`, `ResourceReadCache`, `EvenSplitPerPodTopicQuotaProvider`.

---

### 6.6 Document and validate create-time API input

**Title:** Document and validate optional `messageSizeProfile` on topic create API

Field exists on `TopicResource` but is undocumented in API specs. Clarify optional semantics,
null default, observed intent. E2E: create with profile → GET returns it; create without →
null.

**Acceptance criteria:**
- API spec includes `messageSizeProfile` schema.
- E2E coverage for both paths.

**Refs:** `TopicResource`, `VaradhiTopicFactory`, `TopicHandlers`.

## 7. Open Questions

- **Observation algorithm:** fixed window vs EWMA vs reservoir sampling for avg/max on the hot
  path?

- **Multi-pod merge:** how to combine observations from N producer pods before persist?
- **Default size values:** what deployment-default `avg` / `max` bytes are conservative enough
  for dimension coupling without over-allowing?
- **Operator override vs observed:** if both exist, does manual PATCH freeze observation or
  blend?
