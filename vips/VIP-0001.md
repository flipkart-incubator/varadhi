# VIP-0001: Rate Limiting on Message Produce

**Status:** Draft — core design decided; open items tracked in "Open Questions" (tunable
values, coordination wire framing, `produceRegionWeights` validation). Pending review.

## Goal

Varadhi should rate limit message production based on a topic's capacity policy.
When produce traffic to a topic exceeds its configured capacity, the server should
throttle (reject) the excess so a misbehaving producer cannot overwhelm the
messaging backend (Pulsar) or noisy-neighbour other tenants.

## Background / Current State

Findings from the codebase that shape this proposal:

- **Capacity is per-topic, two dimensions.** `TopicCapacityPolicy` carries `qps` and
  `throughputKBps` (KB/s) as the produce-relevant limits (`readFanOut` and
  `retentionPeriodInDays` are about consumption/storage sizing, not produce throttling).
- **A per-topic message-size profile (avg / max) is tracked.** Producers observe message size
  and the system persists a per-topic **avg** and **max** (eventually, for long-term traffic
  analysis and capacity planning — the `qps ↔ bandwidth` relationship). The rate-limiter design
  uses this profile to couple the two capacity dimensions (see "Dimension coupling"), which
  removes most of the large-message bookkeeping below.
- **The produce hot path already enforces a hard max message size.** `MessageRequestValidator`
  rejects any message above `maxRequestSize` (default 5 MB) with **HTTP 413** *before* the
  capacity check runs — so a single message is already bounded, independent of the limiter.
- **The throttle *response* is already wired, but no limiter exists.**
  `ProduceStatus.Throttled` maps to HTTP 429 in `ProduceHandlers`, and
  `flow.produce.message-to-topic` documents a "capacity check / 429 over-capacity"
  step — but `ProducerService.produceToValidTopic` performs **no** capacity
  enforcement today. This is a documented-but-unimplemented gap.
- **varadhi-server is a horizontally-scaled fleet.** A topic's produce traffic is
  spread across many `roles:[Server]` nodes, while the capacity policy is a single
  per-topic number. The core challenge: how N nodes collectively honour one
  per-topic quota.
- **A consumer-side throttler already exists** (`SlidingWindowThrottler` / `Throttler` /
  `ThresholdProvider`). It throttles *delivery* on error-rate (a different problem) and is a
  *delaying* throttler that needs a background scheduler. We evaluated reusing it but chose a
  daemonless token bucket instead (see Design); it remains a useful reference.
- **The controller can host coordination, but the existing fan-out is NOT reusable.**
  `varadhi-controller` manages membership/assignments and is the natural coordination host.
  However the existing entity-event mechanism (`ResourceEventProcessor` / `MessageExchange` /
  `ResourceEventDispatcher`) is built for **rare metastore changes** and does not fit RL
  (verified in code):
  - `MessageExchange.publish()` is unimplemented; fan-out is a **controller->node unicast
    loop** of per-node `request()`s.
  - It is **one-directional** (no node->controller path) and broadcasts an **identical
    payload** to all nodes with **all-nodes-ack-before-commit** + unbounded retry — the
    opposite of per-pod-distinct shares at a 5s cadence.
  - Per-node queues are **unbounded `LinkedBlockingQueue`s** processed serially — a stuck
    node causes head-of-line buildup; RL needs **bounded, coalescing (latest-wins)** queues.
  - The wire is **JSON, double-serialized** (`ClusterMessage.payload` is a JSON string,
    re-serialized by `MessageExchange`) — the binary bandwidth numbers below are not
    achievable on this path.
  RL coordination is therefore a **new bidirectional protocol** (see Design), reusing the
  controller as host and the Vert.x event bus as substrate — but not `ResourceEventProcessor`.

## Requirements & Constraints (decided)

- **Primary intent: protection / guard-rail.** Stop runaway producers from
  overwhelming the backend / other tenants. Approximate enforcement is acceptable;
  strict billing-grade quota accuracy is a non-goal for v1.
- **Breach behaviour: fail-fast.** Reject over-quota produce immediately with
  HTTP 429 (`ProduceStatus.Throttled`). No in-process queuing/delaying on the
  non-blocking produce hot path.
  - `Retry-After` hint header is an optional nice-to-have, **not implemented in v1**. (It is
    HTTP-only; gRPC has its own retry-hint conventions. Deferred until there's a concrete need.)
- **Traffic distribution may be uneven.** HTTP/2 (and planned gRPC-over-HTTP/2)
  connection pinning at the load balancer can send most of a topic's traffic to a few
  nodes. The design must **not** hard-assume even distribution across the fleet.
- **Dimensions: enforce both.** A message consumes 1 `qps` permit + its byte size;
  throttle if **either** `qps` or `throughputKBps` is exceeded.
  - **Byte size = total message size** — payload **plus** all header keys and values
    (`Message.getTotalSizeBytes()`), not just the payload — since that is what the backend
    and bandwidth actually carry.
- **Scope is per-topic, not per-producer (acceptable).** Quota is enforced at topic
  granularity, so it does not isolate one client from another *within* a topic. This is fine
  for Varadhi: a topic has a single owner, so a multi-tenant/multi-producer topic is rare and
  generally an anti-pattern. Per-producer fairness is explicitly out of scope.
- **Failure mode: fail-open with a static floor.** If the coordination mechanism is
  unavailable, keep accepting produce but enforce a conservative static per-pod cap
  (`regionBudget ÷ podCount × (1 + fallbackBuffer)`) as the floor of protection — fully
  computable from data the pod already holds locally (see Failure handling).
- **Buffer is primarily a fallback knob.** Two independent, configurable buffers:
  - `fallbackBuffer` (default ~0.25): applied to the static even split. Justification: an
    even split under-serves a skewed/pinned pod; biasing toward *allow* keeps the degraded
    floor catching only egregious overload, not normal skew.
  - `normalBuffer` (default 0): optional tolerance in coordinated mode. Coordinated shares
    already absorb growth via unused-budget redistribution, so this defaults off; enable
    only as a deliberate lean-to-allow / intra-interval growth margin.

## NFR / Scale (decided)

| Dimension | Value | Design impact |
|---|---|---|
| Active topics / regional cluster | 1,000 - 10,000 | Per-pod limiter state and coordinator state must stay small per topic; ~10k entries per pod is fine (tens of MB). |
| Server pods / regional cluster | 10 - 50 (moderately dynamic) | Static `quota ÷ N` is coarse; coordinator fan-out to 10-50 pods is cheap. |
| Per-topic QPS | Mixed: most < 1k qps, a few hot topics 10k-100k | Low-QPS topics split across 10-50 pods round to near-zero per pod under static split -> over-throttling. Adaptive shares (concentrate quota on the few pods actually receiving traffic, esp. with HTTP/2 pinning) handle this well. |
| Aggregate produce QPS / cluster | 100k - 1M | A per-message network hop to a central limiter is too expensive -> enforcement must be a **local in-memory** check; coordination must be **out-of-band** (not per message). |
| Added produce latency budget | A few ms (p99) | Local check is sub-microsecond; comfortably within budget. Rules out synchronous per-produce coordination. |
| Quota scope | **Global across regions** | A topic's capacity is one number shared across regions. Synchronous cross-region coordination on the hot path is infeasible (inter-region latency >> few ms). Implies a **hierarchical split: global quota -> per-region budget -> per-pod share**, where the global->region split is coarse and updated slowly. |

### Implications

- Enforcement is a **local in-memory** decision on every pod (no network hop on the hot path), at 100k-1M aggregate qps.
- Coordination (share computation + distribution) is **out-of-band and asynchronous**, at a much lower rate than produce.
- The "global" scope makes this a **two-level** problem:
  1. **Global -> region**: divide the topic's global quota into per-region budgets (coarse, slow-changing).
  2. **Region -> pod**: divide a region's budget across that region's pods (the per-region controller's job; this is the Approach B mechanism).
- Static `quota ÷ N` is unsuitable as the *primary* mechanism for the many low-QPS topics, but remains a valid **fail-open fallback floor**.

### Global -> region split (decided for v1)

- The global quota is divided into per-region budgets using a **per-topic configurable
  region weight**: `regionBudget(topic) = globalQuota(topic) × regionWeight(topic, region)`.
- Weights are **static config in v1**, but the region→pod mechanism consumes only the
  resulting per-region budget — so a later phase can replace static weights with a
  **dynamic global weight coordinator** without changing the rest of the design.

## Approaches Considered

### Approach A — Local-only static split (`quota ÷ N`, with buffer)
Each node enforces `regionBudget ÷ podCount × (1 + fallbackBuffer)` locally; no per-topic coordination.
- **Pros:** simplest; zero coordination/RPC; lowest hot-path cost.
- **Cons:** does not solve skew — a node a client's HTTP/2 connection is pinned to gets
  only `1/N` of the budget and 429s even while the topic is globally under quota.
- Effectively the *degraded mode* of Approach B.

### Approach B — Local enforcement + adaptive shares from a coordinator (recommended)
Each node enforces locally with an in-memory limiter, but its per-topic *share* is
recomputed periodically by the controller based on each node's observed demand and
pushed back over the event bus. Falls back to Approach A's static split when
coordination is unavailable (fail-open).
- **Pros:** adapts to skew; approximate but accurate enough; fail-open fallback built in;
  reuses the controller as coordination host (but needs a new bidirectional protocol — the
  existing entity-event fan-out is unsuitable).
- **Cons:** redistribution interval adds reaction latency; controller holds per-active-topic
  share state; more moving parts.

### Approach C — Shared distributed counter (global token bucket)
A global per-topic counter in a fast shared store; every produce decrements it.
- **Pros:** most globally accurate.
- **Cons:** Varadhi has only Pulsar + ZK (ZK can't take per-message counter writes; no
  Redis-class dependency); adds a network round-trip on the produce hot path; new infra.
  Over-engineered for approximate protection.

## Decision

**Approach B**, structured as a two-level hierarchy, with Approach A (static split) as
the fail-open fallback floor. Rationale: it is the only option that satisfies all decided
constraints together — local in-memory enforcement (for 100k-1M qps within a few-ms
budget), out-of-band coordination, skew-tolerance (HTTP/2 pinning), correct handling of
many low-QPS topics, fail-open behaviour, and the global quota scope. It reuses the
per-region controller as coordination **host** but introduces a **new bidirectional
coordination protocol** (the existing entity-event fan-out is unsuitable — see Background).

```
global topic quota
  └─(× per-topic region weight, static config in v1)─▶ per-region budget
        └─(adaptive share by per-region controller, out-of-band)─▶ per-pod share
              └─(local in-memory token-bucket check, qps + bytes)─▶ allow / 429
```

Fallback: if a pod has no coordinator-assigned share (controller down, pod just started,
etc.), it enforces a static `regionBudget ÷ podCount × (1 + fallbackBuffer)` floor, computed
**entirely from data the pod already holds locally** (cached topic quota + cluster membership
+ config) — no controller dependency for the floor.

## Design

### Component placement & enforcement point

- **`varadhi-server.produce-rate-limiter`** (new): holds the per-topic local limiters.
  Invoked by `produce-service` inside `ProducerService.produceToValidTopic`, right before
  producer resolution / publish — the "capacity check" slot already reserved in
  `flow.produce.message-to-topic`. The check is a **synchronous, in-memory `tryAcquire`**
  (no future, no network) so it stays off the event-loop critical path.
- **`varadhi-controller.rate-limit-coordinator`** (new, per region): aggregates per-pod
  demand reports, computes per-pod shares per topic, and pushes shares back over a **new
  dedicated bidirectional channel** (see "Coordination transport"). It reuses controller
  membership and the Vert.x event bus as substrate, but NOT `ResourceEventProcessor`.
- The enforcement decision is **always local**; the controller only *sizes* each pod's
  local budget.

### Local limiter (per topic, per pod) — daemonless token bucket

- **Algorithm: token bucket, compute-on-read** (no background thread). Each `tryAcquire`
  lazily refills from elapsed **monotonic** time (`System.nanoTime()`, **never wall-clock** —
  NTP steps / leap seconds on wall-clock would yield negative or huge refills on the hot path),
  then decides immediately. An idle topic is never touched; nothing to schedule/drain. (We
  deliberately do **not** reuse `SlidingWindowThrottler` — it needs a `ScheduledExecutorService`
  daemon precisely because it *delays* tasks; our breach behaviour is fail-fast, so we never queue.)
- **Two buckets per topic** (throttle if either is in deficit):
  - qps bucket: refill `qps` tokens/sec.
  - bytes bucket: refill `throughputKBps × 1024` bytes/sec.
- **API — admit on positive credit, not cost-coverage:** `boolean tryAcquire(long messageBytes)`
  (qps cost = 1, bytes cost = messageBytes). Admit iff **both buckets currently hold positive
  credit** (`qpsTokens > 0 && byteTokens > 0`); on admit, debit **both** (`qpsTokens -= 1`,
  `byteTokens -= messageBytes`); otherwise consume **neither** and return `false` →
  `ProduceStatus.Throttled` → HTTP 429. The predicate is the *sign* of each bucket, **not**
  whether it can cover the cost — so a message never needs to be smaller than the remaining
  tokens to be admitted. This single rule unifies "both-or-neither" and the large-message case
  (no separate code path). Per-message only — atomic multi-message batch is a non-goal (see
  "Batch / streaming produce").
- **Bytes-bucket capacity ≥ one message by construction (see "Dimension coupling").** Because
  the controller floors a pod's bytes share at `qpsShare × maxMsgSize` (and qps share ≥ 1), the
  bytes-bucket capacity `bytesRate × burstSeconds` is `≥ maxMsgSize`. So a single message
  normally **fits inside the bucket** — the deep-negative "bounded debt" case is the rare
  residual, not the steady state it used to be for low-throughput/large-message topics.
- **Bounded debt (residual case only):** if a bucket is admitted on positive credit and the
  message exceeds the remaining tokens, `byteTokens` simply goes **negative** and subsequent
  produce is throttled until it refills — realizing "a large payload takes multiple windows".
  This is bounded: the pre-RL `maxRequestSize` guard (5 MB, HTTP 413) caps any single message,
  so the bucket floor is `≈ −maxRequestSize` and recovery is `maxRequestSize / bytesRate`.
  - *Concurrency caveat:* with lock-free atomics, several event-loop threads can each observe
    positive credit and debit concurrently, so worst-case deficit is `(concurrent admits) ×
    maxRequestSize`, **not** strictly one message. Acceptable for a guard-rail (a few event-loop
    threads, sub-µs window); if a hard bound is ever required, gate only the rare
    `cost > capacity` messages through a tiny per-topic lock.
- **Burst control:** bucket capacity = `rate × burstSeconds`, `burstSeconds` a small
  tunable (default ~1s, can be sub-second). Token bucket refills continuously, so there is
  no large discrete window; worst-case micro-burst is bounded by `burstSeconds`. Note capacity
  stays **coupled to rate** (the coupling raises the *rate floor*, never decouples capacity from
  it — avoiding a large stored-burst release).
- **Share updates:** the bucket's refill rate = the coordinator-assigned share (or static
  fallback floor); updating it is a cheap volatile write, no hot-path allocation.
- **Concurrency:** one thread-safe instance per topic, shared across all event-loop threads
  via atomics (per-second rates are well within atomic throughput even for hot topics).
- **Memory:** ~10k topics × 2 small buckets/topic = tens of MB per pod — acceptable.

### Dimension coupling (qps ↔ bytes via message size)

The qps and bytes dimensions are **not independent** — they are linked by message size, and
using the tracked per-topic size profile to couple them removes most of the large-message
complexity above.

- **Minimum capping.** `qps` and `throughputKBps` are each floored at **1** — at topic-creation
  capacity planning *and* whenever the controller assigns a pod its share (a pod is never given
  less than the minimum).
- **The qps floor implies a bytes floor.** For a `1 qps` floor to be *honest* (the one message
  it promises actually gets through), the bytes allowance must admit one message's worth of
  bytes. So the effective bytes floor is `max(1 KBps, qpsFloor × msgSize)`; for any topic whose
  messages exceed ~1 KB, the `qps × msgSize` term dominates and the flat `1 KBps` minimum is
  moot. Generalized: a deliverable qps share `q` requires bytes share `≥ q × msgSize`.
- **Two metrics, two uses:**
  - **avg msg size** → sets the *rate* relationship for capacity planning
    (`throughputKBps ≈ qps × avgMsgSize`); operators may set qps + size profile and let
    bandwidth fall out, or set both and have the consistency **validated** (warn/reject if
    `throughputKBps < qps × minMsgSize`). Kept independently overridable, not hard-coupled.
  - **max (or p99) msg size** → sets the *bucket-capacity floor* (`bytesShare ≥ qpsShare ×
    maxMsgSize`) so a single large message fits without going into debt.
- **Effect by regime:**
  - *Low qps, high msg size* (the case that forced bounded-debt): the bytes floor makes capacity
    `≥ maxMsgSize`, so admission is a clean credit check and deep debt is a rare residual. Bytes
    is the binding dimension; qps is slack — correct.
  - *High qps, low msg size:* `bytesRate = qps × smallMsg` is large, so capacity ≫ one message
    and debt never arises; **qps is the binding dimension** (it protects the broker's per-op
    throughput), bytes is slack. The coupling floor is auto-satisfied. No special handling.
- **Both buckets still enforced at runtime.** The coupling is a *provisioning / share-sizing*
  input, not a runtime substitute for one bucket: within a topic, individual messages vary, so
  the bytes bucket must still catch a run of larger-than-avg messages and the qps bucket a flood
  of tiny ones.
- **The controller needs the size profile — new data dependency to implement.** Share
  computation reads `maxMsgSize` (capacity floor) and `avgMsgSize` (consistency); this data must
  reach the controller, which it does not have today. **Decided source:** pods already observe
  per-message size and already send per-interval demand reports, so the report carries the
  topic's observed `avg`/`max` size for the interval, and the controller derives the profile
  from the same reports it aggregates for shares — self-healing, no extra metastore read on the
  coordination path. The **persisted** profile (metastore, on the topic) remains the source for
  topic-creation capacity planning and as the **cold-start seed** for the controller (brand-new
  topic, or controller rebuilt before any report arrives); until either is available, fall back
  to the deployment-default size assumption. This producer→persist→controller plumbing is **new
  work**, not existing.

### Coordination protocol (region budget -> per-pod shares)

All coordination runs **out-of-band**, never per message.

**Only high-throughput topics are coordinated (working-set reduction).** A topic participates
in adaptive coordination only if its `regionBudget` exceeds a configurable
`coordinationThreshold`; the long tail (the bulk of the 1k-10k topics) is **never reported and
never assigned shares**. This follows from the min-cap rationale (small topics barely affect
cluster health) and shrinks the coordinated set from ~10k to tens-hundreds — which is what
makes the coordination transport affordable.
- **Non-coordinated topic** -> each pod independently caps it at the **full `regionBudget`**
  locally. The dominant HTTP/2-pinned case (most traffic on 1-2 pods) is then capped correctly
  at `regionBudget`; the only downside is up to `podCount×` over-allow when such a small topic's
  traffic is *also* spread wide. Bounded in practice by actual pod-spread (pinning keeps it to
  1-2 pods), but **summed across the many small topics this can matter at the backend** — so
  `coordinationThreshold` is operator-tunable (lower it to pull a topic into real coordination)
  and the per-pod long-tail cap can itself be scaled down with pod count.
- **Coordinated topic** -> the adaptive machinery below; its controller-down fallback is the
  conservative `regionBudget ÷ podCount` split (full `regionBudget` per pod would over-allow a
  large topic massively).

**Per-region budget (input):**
`regionBudget(topic) = globalQuota(topic) × regionWeight(topic, region)` (× `(1 + normalBuffer)`
only if a non-zero `normalBuffer` is configured). This is the ceiling for the **sum** of
per-pod shares in the region.

**Pod -> controller report** (every `reportInterval`, default ~5s; only for topics that
**received traffic** this interval — NOT a pre-assigned subset, since any produce can hit any
pod; worst case is all active topics):
- signal = **`accepted` rate (qps + bytes) + a `saturated` flag** (was the pod hitting its
  limit this interval?). Preferred over `attempted = accepted + throttled` because the
  `throttled` term is inflated/gameable by 429 retry storms, whereas a boolean "wants-more" is
  not. The flag is what tells the controller to grow a pod's share.
- The report also carries the topic's **observed `avg`/`max` message size** for the interval, so
  the controller can apply the bytes-share floor (`bytesShare ≥ qpsShare × maxMsgSize`) and feed
  the persisted size profile (see "Dimension coupling"). Two small numbers per reported topic.
- Reports are keyed by **interned int topic-ID** (not topic name) and may be quantized
  (re-send a topic only when its demand moves > X%) to cut churn — see Bandwidth.

**Topic-ID assignment (controller is the authority):** a pod handshakes with the controller,
which assigns each topic a **dense, sequential int ID** (per region). Dense IDs unlock a
positional wire format: in the common dense case, usage/shares can be sent as a plain **array
indexed by ID** with no per-entry ID at all (sparse id+value pairs only when few topics are
active). Exact wire protocol is an implementation detail to be finalized; IDs are soft state
(re-handshake after a controller restart).

**Controller share computation** (feedback / AIMD-style, per topic):
- Each pod's share floors at its reported `accepted` rate; a **`saturated`** pod's share is
  **grown** each interval, and unused share is **reclaimed** from non-saturated pods, bounded by
  `regionBudget`. (This replaces a pure proportional-to-demand split, since the preferred signal
  is `accepted + saturated-flag`, not a demand magnitude.)
- Every actively-producing pod gets a small **minimum per-pod cap** (`minPodShare`) so a pod
  that just started receiving a coordinated topic isn't starved to zero. **`minPodShare` is
  dynamic**, not a fixed constant: it scales down with pod count (and can be tweaked via API)
  to bound over-allocation. Worst-case allowance for a coordinated topic =
  `min(regionBudget × (1 + normalBuffer), podCount × minPodShare)` — state and bound this
  explicitly rather than treating it as a negligible "edge over".
- **Bytes share is floored at `qpsShare × maxMsgSize`** (the dimension coupling). The qps and
  bytes dimensions are not independent — for a pod's qps share to be deliverable, its bytes
  share must admit at least that many max-size messages. Flooring it this way guarantees every
  pod's bytes-bucket capacity `≥ maxMsgSize`, eliminating the deep-debt case at the pod level
  (incl. under the fallback floor, where a tiny `regionBudget ÷ podCount` bytes split would
  otherwise stall a low-throughput/large-message topic for seconds per message).
  - *Cost:* this floor amplifies `minPodShare` over-allocation in the bytes dimension —
    summed across producing pods it is `(active pods for topic) × minPodShare × maxMsgSize`.
    HTTP/2 pinning keeps a low-qps topic on 1-2 pods in practice, but include this term when
    bounding `minPodShare`, and apply the floor only to **actively-producing** pods.

**Demand signal is noisy and lagged — oscillation protection is a toggleable lever, not an
assumption.** The signal is recomputed from demand that is itself shaped by the share, with a ~5s
lag, and HTTP/2 pinning can swing traffic between pods — so oscillation/false-429 is a *possible*
failure mode, not a certain one. Rather than bake control machinery into the day-one path, we
**build EWMA in Phase 2 (it's a small, localized controller-side change) but ship it off, gated by
a per-topic toggle**, and use shadow mode + selective enablement to gauge whether it's needed:
- **Detector metrics (ship first, Phase 2):** per-topic **share churn / interval** and the
  **429 rate on under-budget topics** (`D < regionBudget`). A false 429 on an under-budget topic
  is the smoking gun; if it stays ~0 under prod-representative load, no protection is needed.
- **First lever — EWMA** smoothing of reported demand (controller-side). **Built in Phase 2,
  default-off, flipped per topic** (`oscillationProtection`), so it's an A/B lever you can enable
  on select topics (ideally first in shadow mode) and compare churn / false-429 against a control.
- **Escalate only if EWMA is insufficient:** deadband/hysteresis (change share only when target
  moves > X%) and capped adjustment (`share += α·(target − share)`, α < 1, + max-increase cap).
  These remain **documented, default-off knobs** gated on the detector metrics.

**Controller -> pod assignment** (every interval): each pod's
`topicId -> {qps share, bytes share}` as **deltas** (only shares that changed), plus a
periodic **full resync** (~every 60s / on reconnect) to self-heal missed deltas.
*Why deltas:* once shares converge they change little, so per-interval cost tracks **churn,
not topic count** — this is what makes 10k topics affordable. (Synergy: the high-bandwidth
case — traffic evenly spread so all pods see all topics — is exactly when shares ≈ static and
barely change, so deltas are tiny precisely when the topic set is largest.)

**Lifecycle / staleness (reclaim must not over-throttle a live pod):**
- Reclaim authority is **cluster membership** (ZK session): a pod that truly left is removed
  and its share fully reclaimed next cycle.
- A pod **alive but with missing/delayed reports** (stressed, network) is NOT instantly
  reclaimed — only after K missed intervals, and conservatively. Crucially, **a pod always
  keeps enforcing its last-known share locally** regardless of controller state, so a blind
  controller can never drive a busy pod to zero.
- If the controller wrongly reallocates a quiet-but-alive pod's slice, worst case is brief
  **over-allow** (sum > regionBudget) — which errs safe for a protection guard-rail.
- New active topic on a pod (no share yet) -> locally-computed static fallback floor until the
  next redistribution.
- Topic genuinely idle on a pod -> pod stops reporting; controller reclaims; pod LRU/TTL-expires
  the local bucket to free memory.
- Controller down -> pods keep last-known shares (share TTL); on expiry without refresh,
  recompute the static fallback floor locally. Fail-open throughout.

**Runtime characteristics:** reaction latency ≈ `reportInterval + compute + push` (a few
seconds); buffer absorbs growth between cycles. Controller state is soft (~10k topics ×
≤50 pods, a few MB), rebuilt from incoming reports after a restart.

### Coordination bandwidth (sized, not trivial)

Worst case = a pod sees **all** active topics (no allocation; any produce can hit any pod).

| Encoding | per entry | full report (10k topics) | controller inbound (×50 pods) |
|---|---|---|---|
| topic name (~32B avg) + 2 numbers, JSON | ~60B | ~600 KB/pod | ~30 MB/interval |
| topic name (100B worst) | ~120B | ~1.2 MB/pod | ~60 MB/interval |
| **interned int topic-ID** (4B) + 2 numbers, binary | ~12-16B | ~120-160 KB/pod | ~6-8 MB/interval |

The table assumes 10k topics, but with the **working-set reduction** above only high-throughput
topics (tens-hundreds) are coordinated, so real volume is 1-2 orders of magnitude smaller.
Note the binary row is only achievable on a **dedicated binary codec** (see Coordination
transport) — NOT on the existing `ClusterMessage` JSON path, where entries are ~3-5x larger.
Name-based at a 1s interval (~30 MB/s into a single controller, plus egress) is prohibitive.
Mitigations, all adopted:
- **Working-set reduction** (coordinate only high-throughput topics) — the biggest lever.
- **Intern topics to dense int IDs** (controller-assigned; positional array in the dense case).
- **`reportInterval` ~5s** -> small with the reduced set; well within budget.
- **Deltas + periodic resync** for controller->pod (steady-state egress ≈ 0).
- **Report only topics with traffic this interval**, optionally **quantized** by demand change.

### Coordination transport (new, dedicated)

The coordination path is **new**, not a reuse of `ResourceEventProcessor` (which is
controller->node unicast, one-directional, identical-payload, all-ack, unbounded-queue, JSON
double-serialized — see Background). It must provide:
- **Bidirectional** messaging: node->controller demand reports AND controller->node per-pod
  share assignments (distinct payload per pod) — neither exists today.
- **Bounded, coalescing queues** with latest-wins semantics (only the newest share/report
  matters); never the existing unbounded-retry queue, which would build up at a 5s cadence.
- **Compact binary encoding** via a Vert.x event-bus `MessageCodec` registered for RL messages
  (protobuf / flatbuffers / hand-rolled), bypassing `ClusterMessage`'s JSON-string payload.
  Keeps the Vert.x clustered event bus as substrate without the double-JSON cost.
- **Dictionary epoch guard (correctness requirement, not framing detail).** Dense positional
  ID arrays are only safe if every framed message carries the **dictionary epoch/generation**
  from the handshake; a pod MUST **drop any frame whose epoch ≠ its current handshake epoch**.
  Without this, a frame built under an old ID-space (after the controller rebuilt the mapping)
  would land shares on the **wrong topics, silently**. Epoch is mandatory; exact framing is LLD.
- **Frame-size safety:** confirm the Vert.x clustered event-bus max message size and **chunk
  the periodic full resync** if needed (the working-set reduction makes this unlikely to bite).

If, after the working-set reduction, the coordinated volume turns out small enough, a JSON
encoding may be acceptable as an interim — but the dedicated bidirectional + bounded-coalescing
channel is required regardless of wire format.

### Correctness & availability callouts (decided)

- **Regional failover strands quota (availability regression during incidents).** Static
  `regionWeight` means a surviving region only has its own slice; if a region fails and traffic
  shifts, the survivor 429s while the dead region's global budget sits **stranded** (global quota
  has headroom it can't use until V2 dynamic weights). v1 mitigation: weights are
  **operator-adjustable via UI**, and this is intended to be driven by the **zone-unavailability
  feature** (marking a zone unavailable triggers a weight adjustment). Explicitly called out as a
  known v1 limitation, not silently accepted.
- **`Σ regionWeight` invariant + validation (API concern).** `regionBudget = globalQuota ×
  regionWeight`, so weights must sum to 1 (per topic, across its produce regions) or the global
  quota is silently over-/under-enforced. Validate on write; define **partial-map semantics** for
  `produceRegionWeights` (how unset regions default, and how the "even split" recomputes when a
  produce region is added/removed) so the default path can't drift the invariant.
- **`shareTtl` ↔ controller-failover coupling.** On controller-down, coordinated topics hold their
  last-known share until `shareTtl`, then drop to the conservative `regionBudget ÷ podCount` floor
  — which over-throttles a skewed/pinned pod. Since a **stale share is safer than the floor**,
  `shareTtl` should be sized **≥ worst-case controller downtime/rebuild** (measured, not arbitrary);
  err long. Reclaim of *departed* pods is membership-driven and independent of `shareTtl`, so a
  long TTL does not delay that.

### Batch / streaming produce

- **Streaming is the model.** HTTP/2 streaming produce is a sequence of individual messages on
  one stream: the limiter does per-message `tryAcquire(bytes)` -> **works as-is**; throttling
  mid-stream yields per-message 429 / stream backpressure. This gives per-message control and
  covers what "batch over HTTP/2" needs.
- **Atomic multi-message batch (one request = N messages) is a non-goal for v1.** A large
  all-in-one payload can't sensibly be acked within a tight latency budget (we won't accept a
  1MB blob in ~50ms); such volume should flow as a stream and be paced across windows.
- **Large single messages** are handled by the dimension coupling (capacity ≥ `maxMsgSize`) plus
  the residual bounded-debt rule above, not by a batch API.

### Configuration & entities

- **Topic-level:** the **global quota** stays in `TopicCapacityPolicy` (`qps`,
  `throughputKBps`) — unchanged. The **per-region weight** does NOT go there (that class is a
  flat, region-agnostic value object reused by storage sizing and consumer assignment).
  Instead add a region-keyed `Map<String, Double> produceRegionWeights` on `VaradhiTopic`,
  parallel to its existing region-keyed `internalTopics` map; **default = even split across the
  topic's produce regions** when unset. (Static config in v1.)
- **Message-size profile (avg / max).** Persisted per topic (producers observe; system persists
  eventually) for traffic analysis and capacity planning, and consumed by the **dimension
  coupling**: `avg` informs the `qps ↔ throughputKBps` consistency relationship, `max` (or p99)
  sets the bytes-bucket capacity floor (`bytesShare ≥ qpsShare × maxMsgSize`). Until a topic has
  a profile (brand new), fall back to the deployment-default size assumption.
- **Minimum capacity capping.** `qps` and `throughputKBps` are each floored at **1** — enforced
  at topic-creation capacity planning *and* in every controller share assignment (a pod is never
  given a share below the minimum). The effective bytes floor is `max(1 KBps, qps × msgSize)`
  (see "Dimension coupling"); validate at creation that `throughputKBps` is consistent with
  `qps × msgSize` rather than an arbitrarily smaller value.
- **Per-topic rollout controls (on `VaradhiTopic`):**
  - **`rateLimiterMode: disabled | shadow | enforced`** — the rollout lever. `disabled` =
    limiter does nothing. **`shadow`** = the full pipeline runs (buckets, reporting, coordination,
    share assignment, would-it-throttle decision) but **never returns 429** — all produce is
    allowed and "would-have-throttled" metrics are emitted instead, so quota sizing *and*
    oscillation can be validated on real traffic before enforcing. `enforced` = actually 429s.
    Roll a topic `disabled → shadow → enforced`. Default at rollout: `disabled` (operator
    opt-in per topic), with a deployment-level `defaultMode` for new topics.
  - **`oscillationProtection: on | off`** (default `off`) — per-topic toggle for EWMA demand
    smoothing, so it's an A/B lever (enable on a candidate topic, compare churn / false-429 vs a
    control). Best exercised first in `shadow` mode.
- **Deployment-level `RateLimiterOptions`:** `enabled`, `defaultMode` (default mode for topics
  with no explicit `rateLimiterMode`; safe default `disabled`/`shadow`), `coordinationThreshold` (regionBudget
  above which a topic is adaptively coordinated; below it, local-only at full regionBudget),
  `fallbackBuffer` (default 0.25), `normalBuffer` (default 0), `burstSeconds` (default ~1),
  `reportInterval` (default ~5s), `resyncInterval` (default ~60s),
  `shareTtl` (size ≥ measured worst-case controller failover, not arbitrary),
  `missedReportGraceIntervals` (K), `minPodShare` (**dynamic** — scales down with pod count;
  API-tweakable), `demandQuantizationPct` (report-churn threshold), local-bucket idle-expiry TTL.
- **Damping knobs (see share computation):** `demandEwmaIntervals` (EWMA — built in Phase 2,
  default-off, flipped per topic via `oscillationProtection`); `shareDeadbandPct` (hysteresis) and
  `shareAdjustAlpha` + `maxShareIncreasePerCycle` (capped adjustment) — **default-off**, gated on
  the detector metrics, only if EWMA proves insufficient.
- **Kill switch:** `enabled=false` (deployment-wide) disables enforcement entirely (limiter allows
  all) — important for safe rollout given this is a new hot-path component. The per-topic
  `rateLimiterMode` is the finer-grained complement (a single topic can be `shadow`/`disabled`
  without flipping the global switch).

### Failure handling (summary)

| Condition | Behaviour |
|---|---|
| Controller / coordinator down | Pods keep last-known shares; after `shareTtl`, **locally recompute** the static `regionBudget ÷ podCount × (1 + fallbackBuffer)` floor from cached topic quota + cluster membership + config. Fail-open, no controller dependency. |
| Pod just started / topic newly active | Locally-computed static fallback floor until first coordinated share arrives. |
| Pod alive but reports missing/delayed | Not reclaimed for K intervals; pod keeps enforcing last-known share locally; worst case is brief over-allow (errs safe). |
| Limiter bug / emergency | `enabled=false` kill switch bypasses all checks; or set a single topic's `rateLimiterMode=shadow`/`disabled`. |
| Over quota, topic in `shadow` mode | Allowed (no 429); **would-have-throttled** metric incremented for validation. |
| Over quota, topic `enforced` (normal) | 429 `Throttled` (fail-fast); optional `Retry-After` for HTTP. |

### Observability (decided)

**Per-pod, per-topic** (via existing produce telemetry -> otel-collector): accepted qps,
accepted bytes-rate, rejected qps, rejected bytes-rate, plus current assigned share (qps + bytes),
and the topic's current `rateLimiterMode`.

**Shadow-mode metrics:** **would-have-throttled** qps + bytes-rate per topic (decisions the limiter
*would* have made were it enforced) — the primary signal for sizing quotas and gauging
oscillation/false-429 before flipping a topic to `enforced`.

**Controller (coordinator):** redistribution cycle latency, topics/pods tracked, reclaim/assign
events, coordination bandwidth, and — most useful operationally — the count/list of topics
currently **demand-saturated** (`D > regionBudget`, i.e. actually being throttled) and aggregate
region-budget utilisation.

**Damping detector metrics (ship in Phase 2, before any damping):** per-topic **share churn /
interval** (how much shares move) and the **429 rate on under-budget topics** (`D < regionBudget`)
— a false 429 on an under-budget topic is the smoking gun that demand noise/lag is causing
oscillation. These metrics drive the evidence-based decision to add EWMA (then, only if needed,
deadband/capped-adjustment).

**Cardinality is the real risk** (10k topics × ~5 series × 50 pods ≈ millions of series).
Decided strategy:

- **Tiered routing via two otel-collector pipelines** (routing/aggregation lives in collector
  config, not RL code):
  - **Self-managed Prometheus, short retention** (hours/days): full-cardinality per-topic,
    per-pod series for drill-down/debugging.
  - **Central APM, long retention**: aggregated series with the **pod dimension summed away**
    and rolled up to project/tier/region for dashboards/alerting.
- **Aggregate the pod dimension at the collector** before APM export (~50x reduction).
- **"Interesting topics" detail, two complementary modes:**
  - *Dynamic:* the limiter auto-emits full per-topic detail when a topic is saturated/near-limit
    (no config needed).
  - *Static:* a per-topic opt-in flag to always emit full detail for known high-value topics.
- **Per-topic emit interval: 60s** (system metrics stay finer-grained).
- **Consistent tags required on every metric** — org / project / region / topic / tier — so the
  collector can roll up for APM and operators can drill down in Prometheus. This is the only
  code-side requirement; the rest is collector/deployment config.

### Testing strategy

- **Unit:** token-bucket `tryAcquire` (refill math via **monotonic clock**, **credit-based
  admission** — admit on `both > 0`, debit both, never cost-coverage — burst bound, share
  update); **dimension coupling** (bytes share floored at `qpsShare × maxMsgSize` ⇒ a single
  max-size message fits without debt; deep bounded-debt only as the residual case, bounded by
  `maxRequestSize`); coordinator allocation
  (feedback/AIMD split from `accepted + saturated-flag`, minimum floor, reclamation on
  idle/crash); **EWMA** smoothing converges without oscillation under noisy/lagged demand and is a
  no-op when `oscillationProtection=off`; **shadow mode** computes a throttle decision but never
  rejects (allows all, increments would-have-throttled);
  reclaim-safety (missing reports without membership loss do NOT zero a live pod's share); local
  fallback computation from cached quota + membership + config; **epoch guard** (frames with a
  stale dictionary epoch are dropped, never misapplied); `produceRegionWeights` validation
  (sum-to-1, partial-map defaults); dynamic `minPodShare` worst-case bound.
- **Integration / E2E:** sustained over-quota produce yields 429 and recovers when traffic
  drops; skewed traffic (most load on one pod) is not over-throttled while globally under
  quota; controller-down recomputes static floor locally without dropping to zero; HTTP/2
  streaming produce is throttled per-message correctly; large single message accepted via
  bounded debt then paced; **a `shadow`-mode topic over quota is never 429'd but reports
  would-have-throttled**; **per-topic `oscillationProtection` toggle changes behaviour for only
  that topic**; kill switch disables enforcement.
- **Load / bandwidth:** validate coordination traffic at 10k topics × 50 pods stays within
  budget (int-ID + deltas + ~5s interval).

## Phasing / Rollout

1. **Phase 1 — local enforcement only (foundation, not prod-sufficient):** daemonless
   token-bucket limiter in `produce-rate-limiter`, fed by the **static fallback** share
   (`regionBudget ÷ pods`), behind the `enabled` kill switch. Ships protection immediately
   (Approach A) and validates the hot-path cost + 429 wiring, but the static split over-throttles
   skewed/HTTP-2-pinned traffic — so it is a stepping stone, **not expected to hold real
   production workloads on its own**.
2. **Phase 2 — adaptive coordination (the prod-grade mechanism):** add `rate-limit-coordinator`,
   the **new dedicated bidirectional coordination channel** (bounded coalescing queues + binary
   `MessageCodec`, not `ResourceEventProcessor`), `accepted + saturated-flag` reporting, and
   adaptive shares for high-throughput topics (full Approach B). Region weights static. Ships the
   **per-topic rollout controls** — `rateLimiterMode` (`disabled → shadow → enforced`) and the
   `oscillationProtection` toggle — plus the **shadow-mode** "would-have-throttled" metrics and the
   **damping detector metrics** (share churn, false-429-on-under-budget). **EWMA is built here**
   too (small change) but ships **off**, flipped per topic; shadow mode + selective enablement on
   real, production-representative traffic is how we gauge whether oscillation protection is needed
   (and, only if EWMA proves insufficient, escalate to deadband/capped-adjustment).
3. **Phase 3 (future / V2) — scale & global coordination.** Deferred extensions that the v1
   design intentionally leaves room for, to be done only if needed:
   - **Dynamic global weights:** replace static `regionWeight` with a global coordinator that
     demand-weights region budgets; region/pod machinery unchanged.
   - **Sharded coordinator:** if a single per-region controller becomes the coordination
     bottleneck (report/compute/fan-out load), shard coordination by **hashing topic ->
     controller shard**, so each shard owns a disjoint topic set. Orthogonal to the local
     enforcement path (pods still enforce locally; only the coordination owner changes).
